# Hardware Behaviors and Limitations

This document describes the behavior of the accelerator as it actually executes, not as a user of a typical ML framework would assume it executes. It is written as prose rather than a reference table because most of the constraints below are not independent — they compose, and the consequences only make sense when read together. When bringing up a new model, the reader should treat every paragraph below as a precondition that the model's forward pass must be rewritten to satisfy before any kernel work begins; failure to satisfy any one of them will either silently produce wrong results or make the operation inexpressible in the instruction stream.

## Compute width: the 64-ALU floor

The datapath contains 64 ALUs operating in lockstep, and this width is not a performance hint — it is a hard minimum. Every call into a core function must present at least 64 elements along the innermost processed dimension, and in practice every call should present a multiple of 64 elements along that dimension, because the hardware has no native concept of a partial lane. If a model's natural tensor shape is narrower than 64 along the axis the operation consumes (a head dimension of 32 or 48, a hidden size that is not a multiple of 64, a vocabulary reduction over fewer than 64 classes, a trailing tile at the end of a non-aligned dimension), the model code is responsible for padding the tensor up to the next multiple of 64, tracking which lanes are real and which are padding, and masking or discarding the padded lanes on the output side. The hardware will not do this; it will happily compute on the padding and return values for lanes that the model must then ignore. This is the single most frequent source of silent numerical bugs during bring-up, and it is why almost every model file in this repository contains ad-hoc padding and slicing logic around otherwise ordinary ops. ## Memory hierarchy and the DRAM budget

The device has 4GB of DRAM, and the entire working set of the model — parameters, activations, and the compiled instruction stream itself — must coexist within that 4GB. The DRAM is partitioned into regions for each of these three classes of data; there is a default partitioning that is sensible for a generic transformer, but the partitioning is not fixed and should be adjusted per model when the defaults do not match the model's actual memory profile. A model with unusually large activations (long context, high resolution vision, long audio) will need more activation space at the expense of parameter space, which in turn constrains how much of the weight set can be resident at once and therefore how the weight streaming is scheduled. A model with an unusually large instruction stream (many unique shapes, many distinct fused-op variants, long unrolled decode loops) will need more instruction space, and if the compiled program does not fit, the only remedies are to reduce shape diversity, increase reuse of compiled instruction blocks, or repartition away from params/activations. Treat the 4GB as a single shared budget that must be balanced per model, not as three independent pools.

## SRAM: two 512KB banks with a fixed role convention

On-chip SRAM is split into two banks, URAM A and URAM B, each 512KB. The convention — and the convention is load-bearing for every existing kernel and every existing model file — is that URAM A holds activations and URAM B holds weights. Following this convention is what allows the matmul and elementwise kernels to issue their reads from the two banks in parallel without contention; deviating from it is possible but requires rewriting the access pattern of every op that touches both operands, so in practice new model code should keep activations in A and weights in B and only reconsider if the model has a structural reason (e.g. a weight-on-weight operation, or an activation-on-activation matmul where one activation must be treated as the "weight" side of the kernel).

## SRAM row granularity: the 128-byte access quantum

This is the hardware behavior that most often forces a model's code to look strange, and it is the one that new contributors most often forget. The SRAM is physically organized as rows of 128 bytes — exactly 64 bf16 elements per row, aligning with the 64-ALU width — and the SRAM cannot be addressed at sub-row granularity. There is no "read bytes 32..96 of row N", there is no striding that crosses row boundaries at a non-row-aligned offset, and there is no notion of a partial-row transfer. Every SRAM read and every SRAM write moves an entire 128-byte row. A consequence that is not immediately obvious: any logical tensor whose innermost dimension is not a multiple of 64 bf16 elements must be laid out in SRAM with each "row" of the tensor occupying its own physical row (or its own integer number of physical rows), padded out to the 128-byte boundary, because there is no way to pack two short rows into one physical row and then address them separately later. The padding is not just a compute-side concern; it is baked into the memory layout.

The canonical bring-up pain point for this rule is RoPE on a model with a head dimension under 64. RoPE operates on a high/low split of each head, so a head dimension of, say, 48 would logically produce two halves of 24 elements each. Because neither half can share a physical row with the other (they must be independently addressable to be rotated against different cos/sin tables), each 24-element half gets its own 128-byte row, of which 48 bytes are real data and 80 bytes are padding, for every head, for every token. The model's RoPE implementation must therefore allocate, address, and mask this layout explicitly, and must feed the 64-ALU datapath a padded-to-64 operand while tracking that only the first 24 lanes are meaningful. Every operation downstream of RoPE that consumes those halves must agree on the same padded layout. This pattern — "head dim below 64 forces padded per-row layout for any op that needs a sub-head split" — recurs outside RoPE (grouped attention variants, gated activations with a split, any op that conceptually splits a narrow innermost dimension) and should be recognized on sight during model bring-up.

## Alignment is the model's problem, not the hardware's

The combined effect of the 64-ALU minimum and the 128-byte SRAM row is that model shapes almost never line up natively with the hardware, and the obligation to make them line up falls entirely on the model code. Hidden sizes, intermediate sizes, head dimensions, vocabulary sizes, sequence lengths, batch dimensions, expert counts, image patch counts — any of these that is not already a multiple of 64 must be padded in the model's forward pass, the padding must be carried through every subsequent op, and the padding must be stripped (or masked to zero, depending on the op) before any result is compared to a reference implementation. The existing per-model helper functions in this repository exist almost entirely to handle this alignment burden; they are not elegant abstractions, they are workarounds for the fact that the hardware refuses to do anything with fewer than 64 lanes at a time and refuses to address less than a full row at a time.

## Execution model: compile, then execute

The device is not operated imperatively. The convention in this repository, and the convention the rest of the toolchain assumes, is that the model's forward pass is first fully *compiled* — every op is emitted into an instruction chain with its operand addresses, shapes, and fused-op flags resolved — and then the resulting instruction stream is *executed* in a single handoff to the hardware. Individual ops are not dispatched one at a time from the host. The practical implications are that any decision that depends on a runtime value (a dynamic sequence length, a variable number of active experts, a sampled token id) must either be baked into the compiled program as a fixed-shape, possibly over-allocated variant, or handled by compiling multiple specializations and selecting among them at execute time. Control flow that branches on activation values is expressed by running all branches and masking, not by skipping work. This is also why shape diversity costs instruction-DRAM space: every distinct shape signature a model uses is a distinct compiled block.

## What to check first when bringing up a new model

Before writing any kernel, walk the model's forward pass and, for every tensor and every op, answer: is the innermost dimension consumed by this op a multiple of 64, and if not, where does the padding come from and where does it get stripped; does any op split a dimension smaller than 64 into sub-halves, and if so can the two halves tolerate living on separate padded SRAM rows; does the total params + peak activations + estimated instruction stream fit in 4GB under some partitioning, and if not which of the three is the one to shrink; does the op schedule keep activations in URAM A and weights in URAM B, or does the model have a structural reason it cannot; and is every runtime-dependent shape either fixable at compile time or expressible as a finite set of compiled specializations. A model that passes all five of these checks is one that can be brought up by writing kernels; a model that fails any of them is one that needs its forward pass rewritten before kernel work starts.
